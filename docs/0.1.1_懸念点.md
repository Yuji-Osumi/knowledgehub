# 0.1.1 追加検討事項・懸念点

## 1. 要件の肥大化・MVPの範囲
- 機能が多くなるとMVPが膨らむ可能性
- **懸念:** 「階層構造＋タグ＋関連ノート＋全文検索＋無料AI補助」などで開発負荷増
- **対応:** 最初は必須機能に絞り、拡張機能は後回し

## 2. 技術面での実装負荷
- FastAPI + SQLAlchemy + PostgreSQL + Next.jsのフルスタックは学習コストが高い
- **懸念:** 階層構造（再帰CTE）や全文検索・バージョン管理を同時に実装すると複雑
- **対応:** MVPでは記事作成・編集・階層・検索を優先

## 3. UI/UX
- Markdown主体の軽量UIは好みだが、操作性の確認が必要
- **懸念:** 階層構造や関連ノートが増えると見やすさが低下
- **対応:** UIプロトタイプを作成して操作性を早期確認

## 4. データ量・性能
- MVP段階では少量データでも十分だが、将来的に1万件以上扱う場合の性能確認が必要
- **懸念:** 複雑なフィルタや重み付け検索で遅延が出る可能性
- **対応:** ベンチマークテストをMVP段階で実施

## 5. 競合との差別化
- NotionやObsidianなどの強力な競合が存在
- **懸念:** 差別化ポイント（軽量・Markdown中心・無料AI補助）がユーザーに伝わるか
- **対応:** サンプル画面やユースケースを用意して体験を具体化

## 6. AI連携
- 無料AIでの関連ノート提示は便利だが精度やUXに課題
- **懸念:** LLMを使わない場合、関連度算出が難しい可能性
- **対応:** MVPではタグ・タイトル類似度ベースで簡易実装し、後からEmbeddings等で拡張
