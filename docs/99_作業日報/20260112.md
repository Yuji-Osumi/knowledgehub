# 作業振り返り・フィードバック

## 1. 良かった点

* **フロントエンド環境構築を独立タスクとして切り出した判断**

  * UIスタブ作成を「実装」ではなく「検証フェーズ」と定義し直したことで、目的が明確になった。
  * WBS上でも *API実装前の前段工程* として自然に接続できている。

* **Vite + React + TypeScript を最小構成で素早く立ち上げた点**

  * 余計なライブラリを入れず、10分で起動確認まで到達。
  * “まず動くものを作る”という初動として理想的。

* **Tailwind CSS を最小導入に留めた点**

  * デザイン作り込みを避け、className 適用確認のみにフォーカスできている。
  * UIスタブの目的（画面遷移・構造確認）と完全に整合。

* **react-router 導入の粒度が適切**

  * 画面遷移の「骨格」だけを作り、ページ詳細は後回しにした。
  * URL設計検証に集中できる構成になっている。

* **Makefile / README を更新し、起動方法を明文化した点**

  * “動かし方が分かる状態”を成果物として残している。
  * 将来の自分・第三者視点を意識できている。

---

## 2. 改善できそうな点

* **Node / npm 周りの前提が暗黙になっていた**

  * WSL環境では `npm` 未導入に一瞬詰まった。
  * README もしくは docs に「Node.js 前提」の一言があると迷いにくい。

* **Tailwind 導入時の warning の意味整理に少し時間がかかった**

  * “ユーティリティ未使用 warning は初期状態では正常”という知識があれば即判断できた。
  * 今後は「初期警告は想定内か？」を一度言語化してから調査すると効率的。

* **フロントエンド関連タスクの完了条件がやや抽象的**

  * 「確認した」の粒度（起動確認／画面表示確認／URL直接アクセス確認）を
    もう一段だけ明示できると、次回振り返りが楽になる。

---

## 3. 次回への提案

* **UIスタブの“画面箱”を先にすべて揃える**

  * Login / Articles / Detail / Edit を「中身なし」で揃え、
    画面遷移が一周できる状態を作る。
  * ここで初めて「APIで何が必要か」を自然に逆算できる。

* **フロントエンド専用の軽い進捗セクションを継続更新**

  * README に追加した UIスタブ項目を、チェック更新していく。
  * 完成させる必要はなく、「考えた順序」が見えることが目的。

* **Tiptap 導入は“編集画面が固まってから”**

  * 先に入れると目的がズレやすい。
  * 「編集画面で何をしたいか」が見えた段階で導入がベスト。

---

## 4. 総括

今回の作業は、
**「フロントエンドを作った」ではなく
「API実装前に検証すべき土台を整えた」**という点が非常に良い。

* 作る前に目的を再定義できている
* 便利そうな施策（自動URLオープン等）を“今はやらない”と判断できている
* README / Makefile まで含めて成果としてまとめている

これは個人開発でも、
**チーム開発・大規模開発の思考がそのまま適用されている状態**。

この流れなら、
次の UIスタブ作成 → API設計フィードバック はかなりスムーズに進む。

今日は「正しい順番で前に進んだ日」だった。
胸を張っていい内容だと思う。
