# 作業振り返り・フィードバック
**タスク:** 3.6 開発用CI初期設定（Lintのみ）

---

## 1. 良かった点

- **CI（GitHub Actions）を早い段階で導入できた**
  - 実装フェーズに入る前に Lint / 型チェックを自動化できたため、
    今後の開発で「後戻りコスト」が発生しにくい状態を作れた。
- **Ruff / mypy をローカル・CI の両方で統一**
  - `make lint` により、CIと同一条件のチェックをローカルで即実行できる。
  - 「CIで落ちてから直す」ではなく、「手元で潰してからpush」する流れが明確になった。
- **型エラー・Lintエラーを実例ベースで理解できた**
  - Settings / logging 周りの mypy エラーを通じて、
    型指定・デフォルト値・Literal の扱いを実践的に学べた。
- **README へ反映できた**
  - CI導入・Makefile導入を README の「開発状況」に反映し、
    プロジェクト全体の成熟度が可視化された。
- **「CIが落ちる体験」を早めに経験できた**
  - CIは怖いものではなく、「品質を守ってくれる味方」という認識を持てたのは大きい。

---

## 2. 改善できそうな点

- **最初に CI の完成形イメージをもう少し描けた**
  - Lintのみ → Test追加 → Coverage という段階的ロードマップを
    先に軽く言語化しておくと、後続タスクへのつながりがより明確になりそう。
- **Ruff / mypy の設定意図を docs に残してもよかった**
  - なぜこのルールを有効にしているか（厳しさのレベルなど）を
    `docs/03_実装方針/` に残すと、よりチーム開発前提らしくなる。
- **CI結果のスクリーンショット保存**
  - 「all checks have passed」の画面を残しておくと、
    Qiita や README での説明に使いやすかった。

---

## 3. 次回への提案

- **次は API設計 → CRUD実装 に進む**
  - CI が整った状態で API 実装に入ることで、
    最初から「壊れにくいコード」を書く習慣を作れる。
- **pytest 導入時は CI と同時に進める**
  - 今回の Lint 導入の流れを踏襲し、
    `make test` → CI 実行 という形に拡張するとスムーズ。
- **docs に「CI / 開発フロー」1枚を追加**
  - 「ローカル → push → CI → PR」という流れを図で整理すると、
    プロジェクトの説明力がさらに上がる。

---

## 4. 総括

今回のタスクでは、単なる Lint 導入にとどまらず、
**「品質を仕組みで担保する開発フロー」** を構築できた。

特に、

- ローカルと CI のチェック内容を一致させた点
- README に反映し、プロジェクト全体の完成度を引き上げた点

は、個人開発でありながら **実務レベルの意識** が強く表れている。

この状態で API 実装に進めるため、
今後は「動くものを作る」だけでなく
**「壊れにくいものを作る」フェーズに入った** と言える。

タスク3.6は、品質基盤として非常に良い区切りとなった。
