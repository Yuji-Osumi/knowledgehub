# 0.1 目的・背景定義（詳細版）

## 1. プロジェクトの概要

### 1.1 プロダクト名（仮）
**KnowledgeHub**
個人〜小規模チーム向けに最適化されたナレッジ管理・文書管理アプリケーション。

---

## 2. プロジェクトの目的

本プロジェクト「KnowledgeHub」は、以下の3つの目的を満たすことを目指して開発する。

### 2.1 実用的で継続利用可能なナレッジ管理ツールを構築する
日常的に発生する技術メモ、設計ノート、学習ログが散在しやすいという課題に対し、
「自分が本当に使いたい」と思える快適なユーザー体験を伴ったナレッジ管理アプリケーションを提供する。

- 高速な全文検索によって目的の情報に素早くアクセスできること
- Markdown による快適な記事作成・編集体験
- 階層構造やタグを活用した直感的な整理
- 日々の開発や学習に自然に組み込める操作性

これらの特徴を備えたアプリケーションを構築し、
長期的に自身の知識基盤として利用できることを主目的とする。

### 2.2 設計から実装まで一貫して行えるエンジニアとしての能力を示す
本プロジェクトでは、普段の業務では経験しにくい領域（要件定義、技術選定、アーキテクチャ設計、DB設計、API設計など）を積極的に取り入れる。

特に以下の技術的チャレンジを通じて、
「フルスタックでシステムを構築できるエンジニア」であることを証明する。

- クリーンアーキテクチャ（UseCase, Repository, DI 等）の採用
- FastAPI × SQLAlchemy を用いた Python のバックエンド構築
- Alembic を用いたマイグレーション運用
- PostgreSQL の pg_bigm による高速日本語検索、将来的に Embeddings による意味検索
- Next.js（React）を用いたモダンフロントエンド開発
- Docker / docker-compose による開発環境構築

これらを通じて、アプリケーション開発の全工程を理解し、設計力と技術力をアピールする。

### 2.3 新しい技術・アーキテクチャの習得を通じて自身のエンジニアリング成長につなげる
普段携わらない領域に挑戦することで、新たな技術的視野を獲得し、
長期的なキャリア形成に必要なスキルセットを広げる。

主に以下のスキル習得を目的とする。

- API・DB設計の基礎〜応用（ER 図、正規化、N:N モデル、階層構造）
- ログ・バージョン管理など運用寄りの設計思考
- ユーザー体験を重視した UI/UX 設計
- 小規模サービスを自力で構築・改善し続けるためのフルスタックスキル

これにより、業務では得られない包括的な開発経験を蓄積し、
エンジニアとしての成長に直結するプロジェクトとする。

## 3. 背景・課題（なぜこのアプリが必要なのか？）

### 3.1 背景
- 日々の学習や読書、ゲームなどの情報を記録したいが、従来のノートアプリやWikiでは自由度や操作感が自分に合わない。
- Markdown中心の書きやすさや、VSCodeで書いたときの色付き表示など、軽量で快適な執筆体験を重視したい。
- Notionのテーブルやデータベース管理に感動したが、個人用途では操作が重く、必要な機能だけに特化したアプリが欲しい。
- 既存の競合（Notion、Obsidian、Scrapboxなど）はそれぞれ強みがあるが、個人の思考整理や軽量なMarkdown体験、半自動の構造化・関連付けという点で十分な差別化ができていない。

### 3.2 課題
1. **情報整理の負荷**
   - 書いたメモや記録を後から俯瞰・検索するのが手間。
   - 手作業でタグやカテゴリ分けをするのは時間がかかる。
2. **操作性・速度の課題**
   - Notionなど既存ツールは機能が多い反面、動作が重く、サクサク書ける体験がない。
3. **構造化の不足**
   - Markdown主体で書けるものの、自動でカテゴリ分けや関連ノート提示がされない。
   - 思考マップやノート間の関連性が見えにくい。
4. **個人開発者としての差別化**
   - 既存サービスにはない「軽量でMarkdown中心、AI補助による自動整理」の体験を提供することで、自分が本当に使いたくなるアプリを作れる。
   - 個人開発としての技術アピール（Embeddings活用、無料AIによる自動分類、構造化データ管理）も可能。

### 3.3 まとめ
- **目的:** 個人が毎日使いたくなるようなMarkdown中心の思考整理アプリを作る。
- **課題:** 情報の整理・検索・関連付けの手間、既存ツールの動作の重さ、構造化不足。
- **解決:** 軽量なMarkdownエディタ＋無料AIによる半自動整理・関連ノート推論により、既存ツールでは実現できない快適な情報管理体験を提供。

## 5. 解決したい課題と提供価値

### 5.1 現状の課題
1. **Notionは便利だが重くて複雑**
   - 多機能すぎて使いこなせない
   - 動作が重くサクサク編集できない

2. **Markdownで情報管理したいが体系化が難しい**
   - 階層整理や検索が弱い
   - ノート間の関連性が見えない

3. **趣味ログがバラバラ**
   - 読書、ゲーム、学習が複数アプリに散在
   - 全体像を俯瞰できない

4. **AI連携のコスト問題**
   - 高品質AIは有料
   - MVPでは無料AI＋軽量アルゴリズムで十分

---

### 5.2 提供価値
1. **軽量Notionとしての価値**
   - シンプルUI、軽量動作
   - Markdown主体で快適に書ける

2. **Markdown知識ベース**
   - 自動整理、タグ・カテゴリ付与
   - 関連ノートで知識ネットワーク構築

3. **趣味ログと知識の一元化**
   - 読書、ゲーム、学習を同じUIで管理
   - テーブルビューで俯瞰・フィルタ検索

4. **無料AIによる便利さ**
   - Embeddingsで関連情報表示
   - 高負荷AIは不要で個人でも運用可能

5. **楽しく継続できる体験**
   - 自分のノートが自動でつながる
   - 思考マップで振り返りが楽しい
   - 個人開発者の「使いたくなるアプリ」体験

# 0.1 目的・背景定義（サマリー版）

## 1. プロジェクトの目的
- 個人開発者として自分が毎日使いたくなるナレッジ管理アプリを作る
- Markdown中心で軽量かつ快適に情報整理・検索できる
- Clean ArchitectureやDB設計・検索最適化など、普段経験できない技術をポートフォリオでアピール

## 2. 背景・課題
- 既存サービス（Notion、Obsidian等）は多機能だが動作が重く、個人用途では過剰
- Markdown主体で軽量に書ける知識管理が不足
- 情報の階層化・タグ管理・関連ノート表示など、整理・俯瞰が手間
- 趣味・学習ログを一元管理できるツールがほしい
- 高負荷AIは使わず、無料で便利な機能を提供したい

## 3. ターゲットユーザー
- 個人開発者：学習・技術メモを効率的に整理したい
- 趣味記録者：読書・ゲームログなどを軽量に管理したい
- フリーランス：案件情報やコードスニペットを個人で整理したい

## 4. ユースケース例
- 読書メモ・学習記録をMarkdownで作成
- ゲームプレイ記録をテーブル形式で整理
- 技術メモや仕様書を階層・タグで管理
- 関連ノート提示により知識ネットワークを構築

## 5. 解決したい課題と提供価値
- 軽量でMarkdown中心の快適な執筆体験
- 階層構造・タグ・関連ノートによる情報整理
- 大量データでも高速な検索・参照
- 無料AIによる関連情報提示
- 個人開発者としての技術的アピール

## 6. プロジェクトのゴール（成功基準）
- 自分が日常的に使える完成度
- データ整理・検索がスムーズ
- MVPとして必要機能が完結、将来拡張可能
- 技術的にクリーンな設計が評価できる

## 7. 想定技術スタック
- **バックエンド:** Python + FastAPI + SQLAlchemy + Alembic + PostgreSQL
- **フロントエンド:** Vite + React + TailwindCSS + Tiptap Editor
- **検索・整理:** PostgreSQL全文検索(tsvector + GIN) + 無料AI補助
- **インフラ:** Docker + docker-compose + Nginx、必要に応じてRedis

## 8. 競合・類似サービス比較
- **Notion:** 多機能だが重い → KnowledgeHubは軽量・Markdown主体
- **Obsidian:** ローカル＆リンク型ノート → KnowledgeHubは階層・検索・関連ノートが自然
- **Scrapbox:** リンク中心だがMarkdown非対応 → Markdown対応＆階層整理
- **Zenn/Qiita:** 公開前提で個人管理に柔軟性不足 → 閉じた個人用ナレッジベース
- **差別化ポイント:** 軽量、Markdown中心、階層＋タグ＋関連ノート、無料AI補助、技術的アピール可能
