# 作業振り返り・フィードバック

## 1. 良かった点

- 要件定義を 0.x のWBS構造で分解し、「今どこを考えているか」が常に明確だった
- MVP前提での割り切り（非スコープ・制約）を仕様として明文化できていた
- ChatGPTを意思決定者ではなく、整理・比較・文章化の補助として使えていた
- 「そのまま.mdにして」という進め方により、思考を止めずドキュメントが自然に蓄積された
- データモデル・API・インフラ・テストまで一貫した視点で要件定義を完結できた

## 2. 改善できそうな点

- 0.8（データモデル）と 0.9（API仕様）の間で行き来がやや多く、決定タイミングが曖昧になりやすかった
- UUID採用やTSVECTOR導入などの設計判断について、決定理由を「決定ログ」としてまとめていなかった
- 一部の検討内容（セキュリティや検索方式など）が深掘り寄りになり、MVP基準の線引きに少し迷いが出た

## 3. 次回への提案

- データモデルは「80%で一旦確定 → APIで違和感が出たら戻る」というルールを事前に決めておく
- 設計上の重要判断は、1〜2行の Decision Log として残す（後からの説明・記事化に有効）
- 「これは実装フェーズで考える」と割り切る基準を最初に決め、要件定義で考えすぎないようにする
- 要件定義完了後は、早めに1エンドポイントだけ実装してフィードバックを得る

## 4. 総括

今回の要件定義は、個人開発としては非常に完成度が高く、
**「実装に進んでも迷わない状態」**をしっかり作れた点が最大の成果だった。
完璧を目指さず、判断理由と割り切りを言語化できているため、
この進め方は次の個人開発や実務的な設計作業にもそのまま再利用できる。
